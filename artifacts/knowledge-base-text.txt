Knowledge base — Italian-OSS-Legal-Platform
Questa cartella è un bundle Open Knowledge Format (OKF) v0.1: una directory di file Markdown
con frontmatter YAML, leggibile da persone e agenti AI senza alcun SDK o piattaforma proprietaria.
Convenzioni del bundle
• Un file = un concetto. Ogni documento descrive una singola entità (una fonte, un'entità dati, un
componente, un termine).
• Frontmatter YAML in testa a ogni file. L'unico campo obbligatorio è type; title, description,
resource, tags, timestamp sono opzionali.
• index.md in ogni cartella descrive e indicizza i concetti contenuti.
• Collegamenti tra concetti con percorsi relativi alla radice del bundle, es. [Norma](/modello-
dati/norma.md).
Aree
• Fonti dei dati — fonti normative aperte, formati e condizioni di riuso.
• Modello dati — FRBR, ELI, struttura Akoma Ntoso, entità interne e pipeline.
• Architettura — componenti del sistema e flusso RAG.
• Glossario — termini giuridici e tecnici.

Fonti dei dati
Questa cartella mappa le fonti del diritto utilizzabili da Italian-OSS-Legal-Platform, i formati disponibili
e le condizioni di riuso. È il riferimento per chiunque lavori alla pipeline di ingest.
Concetti
• Normattiva — Open Data — fonte primaria della legislazione statale italiana.
• Gazzetta Ufficiale — pubblicazione ufficiale degli atti.
• EUR-Lex — diritto dell'Unione Europea.
• Giurisprudenza — sentenze e orientamenti in formato aperto.
Principi di trattamento dei dati
• Attribuzione: mantenere e mostrare la paternità.
• Tracciabilità: per ogni documento conservare fonte, URI ELI, formato originario, data di
acquisizione e versione.
• Nessuna alterazione del significato: il testo normativo non va modificato; eventuali
normalizzazioni devono essere reversibili e documentate.
• Aggiornamento: prevedere ri-ingest periodico per recepire modifiche e nuove versioni
(multivigenza).
Normattiva — Open Data (fonte primaria)
Portale: https://dati.normattiva.it/
La fonte principale per la legislazione statale italiana (leggi, decreti legge, decreti legislativi, regi
decreti, codici). Realizzato dall'Istituto Poligrafico e Zecca dello Stato su iniziativa del comitato di
gestione Normattiva.

Esportazioni temporali

Portale: https://www.gazzettaufficiale.it/
Fonte ufficiale di pubblicazione degli atti normativi. Utile per atti recenti, avvisi e per la verifica della

===== PAGINA 7 =====
Modello dati e parsing Akoma Ntoso
Questa cartella descrive come Italian-OSS-Legal-Platform rappresenta una norma italiana a partire
dai dati di Normattiva (Akoma Ntoso / ELI) e come la trasforma in unità interrogabili per la ricerca
semantica e le citazioni.
È una bozza concettuale: lo schema serve a ragionare sui dati, non è ancora un'implementazione.
Concetti di base
• Il modello FRBR — Work / Expression / Manifestation / Item.
• Identificazione: URI ELI — chiave primaria stabile.
• Struttura del documento AKN — articolo e comma.
• Pipeline di trasformazione — da Normattiva all'indice vettoriale.
Entità del modello dati interno
• Norma (Work)
• Versione (Expression)
• Unità (articolo / comma)
• Chunk (ricerca semantica)
• Riferimento (cross-reference)
Il modello FRBR di Akoma Ntoso
Akoma Ntoso adotta il modello FRBR per gestire le numerose derivazioni dei testi normativi
(modifiche, versioni, lingue). Quattro livelli:
Livello Significato Esempio
Work La norma in astratto, in tutte le sue versioni "L. 241/1990"
Expression Una versione specifica nel tempo/lingua "L. 241/1990 vigente al 2020-01-01"
Manifestatio Una rappresentazione in un formato il file XML AKN di quella versione
n
Item La copia concreta (file) il file scaricato e archiviato
Questi livelli sono dichiarati nel blocco <meta> -> <identification> tramite <FRBRWork>,
<FRBRExpression>, <FRBRManifestation>, <FRBRItem>.
Questo modello è il motivo per cui Italian-OSS-Legal-Platform può distinguere "il testo vigente oggi"
da "il testo in vigore nel 2015". Le entità interne Norma e Versione mappano rispettivamente Work ed
Expression.

===== PAGINA 8 =====
Identificazione: URI ELI
Ogni norma è identificata da una URI ELI (European Legislation Identifier) permanente. Italian-OSS-
Legal-Platform usa l'ELI come chiave primaria stabile per collegare versioni, citazioni e riferimenti
incrociati.
Esempio concettuale di pattern ELI:
/eli/it/stato/legge/1990/08/07/241/...
Vantaggi: stabilità nel tempo, interoperabilità europea, deduplicazione affidabile.
Struttura del documento AKN
Un atto (<act>) si articola tipicamente in:
<akomaNtoso>
<act>
<meta> … identificazione FRBR, riferimenti, cicli di vita …
<preface> … titolo, intestazione …
<preamble> … premesse …
<body>
<article> (unità di base: articolo)
<num> (numero articolo)
<heading> (rubrica)
<paragraph> (comma)
<num> (numero comma)
<content> (testo)
…
<conclusions> … formule finali …
L'articolo (<article>) è l'unità fondamentale; il comma (<paragraph>) è la sotto-unità più
rilevante per le citazioni puntuali. Queste unità sono modellate dall'entità Unità.

===== PAGINA 13 =====
• Separazione dati/modello: la qualità dipende dai dati e dal retrieval, non solo dall'LLM.
• Self-hosting possibile: l'architettura deve poter girare interamente sotto il controllo dell'utente.
• Modularità: provider LLM, storage e database intercambiabili.
Frontend (Next.js / TypeScript)
Interfaccia di chat, ricerca, caricamento e visualizzazione documenti con citazioni cliccabili che
rimandano alla fonte (ELI / Normattiva).
Backend / API (Node / TypeScript)
Orchestrazione del flusso RAG (retrieval -> costruzione contesto -> generazione), gestione utenti,
gestione documenti caricati. Vedi il flusso di una domanda.
Indice normativo + Vector DB
PostgreSQL con pgvector (Vector DB) per ricerca semantica e query strutturate sui metadati (filtri
per tipo di atto, vigenza, ecc.). Alimentato dalla pipeline di trasformazione e popolato di Chunk.
Object storage (S3-compatibile: MinIO)
Conserva i documenti caricati dagli utenti e gli artefatti di ingest. Si usa MinIO, object storage open
source compatibile con l'API S3, eseguito in self-hosting tramite Docker Compose insieme al resto
dello stack.
Provider LLM (configurabile)
Almeno un provider a scelta; obiettivo di lungo periodo: supportare modelli locali/self-hosted per
privacy e sovranità dei dati. Il provider genera le risposte nel flusso RAG.
Flusso di una domanda (RAG)
1. L'utente pone una domanda in linguaggio naturale.
2. Il backend recupera i Chunk più rilevanti dall'indice (ricerca semantica + filtri).
3. Costruisce un contesto con i testi normativi e i relativi metadati di citazione.
4. L'LLM genera la risposta citando articolo, comma e fonte (ELI).
5. Il frontend mostra la risposta con i link verificabili alle fonti.
Vedi RAG e citazione verificabile.

===== PAGINA 15 =====
Vigenza
Periodo in cui una norma è in vigore. Una norma può avere più versioni nel tempo. Vedi anche
multivigenza.
Multivigenza
Disponibilità di tutte le versioni storiche di una norma, ciascuna con il proprio intervallo di vigenza. È
fornita da Normattiva ed è fondamentale per il confronto storico.
Testo vigente
Versione di una norma attualmente in vigore (a una certa data). Contrapposto al testo originario.
Testo originario
Versione della norma come pubblicata in origine, prima di modifiche. Contrapposto al testo vigente.
Abrogazione
Cessazione dell'efficacia di una norma, totale o parziale. È uno dei tipi di Riferimento tra norme.
Comma
Suddivisione interna di un articolo; unità tipica per le citazioni puntuali. Nel modello è rappresentato
dall'entità Unità e mappa l'elemento <paragraph> della struttura AKN.
Rubrica
Titolo/intestazione di un articolo. Corrisponde all'elemento <heading> nella struttura AKN.
Recepimento
Atto nazionale che dà attuazione a una direttiva UE. Collega la normativa nazionale alle fonti EUR-Lex
; è uno dei tipi di Riferimento.

===== PAGINA 16 =====
Insieme delle decisioni dei giudici (es. Cassazione, Corte Costituzionale). Come fonte dati vedi
Giurisprudenza.
Akoma Ntoso (AKN / LegalDOCML)
Standard XML internazionale (OASIS) per rappresentare documenti giuridici in modo strutturato, con
metadati. Vedi la struttura del documento AKN e il modello FRBR.
XML NIR (NormeInRete)
Standard XML AgID precedente ad Akoma Ntoso per le norme italiane. Usato come fallback o per atti
storici (vedi Normattiva).
ELI (European Legislation Identifier)
Sistema di URI permanenti per identificare in modo stabile e interoperabile gli atti normativi a livello
europeo. Usato come chiave primaria: vedi Identificazione — URI ELI.
FRBR
Modello (Work / Expression / Manifestation / Item) usato da Akoma Ntoso per gestire versioni, formati
e derivazioni di un documento. Vedi il modello FRBR di Akoma Ntoso e Work / Expression /
Manifestation / Item.
Work / Expression / Manifestation / Item
I quattro livelli del modello FRBR: rispettivamente la norma astratta, una sua versione, una sua
rappresentazione in un formato, la copia concreta. Nel modello dati Norma = Work e Versione =
Expression.
RAG (Retrieval-Augmented Generation)
Tecnica in cui l'LLM genera risposte basandosi su documenti recuperati da un indice, anziché solo
sulla memoria del modello. Permette citazioni verificabili. Vedi il flusso di una domanda (RAG).
Embedding
Rappresentazione vettoriale di un testo, usata per la ricerca semantica. Ogni Chunk ne ha uno.

===== PAGINA 17 =====
Ricerca semantica
Ricerca per significato/concetto, non solo per corrispondenza di parole chiave. Si appoggia agli
embedding e al Vector DB.
Chunk
Porzione di testo (es. un comma) indicizzata per il retrieval, con i metadati di citazione. Vedi l'entità
Chunk.
Vector DB
Database ottimizzato per ricerche su vettori (es. PostgreSQL con pgvector). Nel progetto è l'indice
normativo.
Self-hosting
Esecuzione del software interamente su infrastruttura controllata dall'utente. È un principio
architetturale del progetto (privacy e sovranità dei dati).
Citazione verificabile
Riferimento di una risposta a una fonte ufficiale identificabile (ELI + articolo + comma + data di
vigenza). È il principio cardine del flusso RAG: nessuna risposta normativa senza fonte.
